# Life Memory Vault — 그래프 레이어 기획안 (관계·연결 강화)

작성일: 2026-07-15
작성 배경: 사용자가 검색/lint/enrich 점검 중 "옵시디언 그래프 연결을 활용 못 하고 있다"고 지적, 실측 확인 결과 사실로 확인됨. 이 문서는 "저장·정리는 잘 되지만 결국 키워드 검색 메모앱과 다르지 않다"는 문제를 풀기 위한 기획안이다.
전제: `docs/improvement-plan.md`(P1~P5, ③a~d)의 후속. 그 문서의 CHARTER(`capture-first`/`raw-sacred`/`local-free-first`/`sensitive-by-default`)를 그대로 따른다.

## 1. 문제 정의 — 오늘 확인된 실측 데이터

| 항목 | 실측치 | 의미 |
|---|---|---|
| `40_Notes/Saves` 139개 중 다른 노트/엔티티로의 진짜 상호링크 | 8개 (6%) | 노트끼리 거의 안 엮여 있음 |
| `entity_refs` frontmatter 필드 사용 | 4개 | Wiki Layer 설계(ai-lint.md 2단계)가 사실상 미실행 |
| `40_Entities/*` 실제 파일 | 5개 (2026-06-07~10, 대부분 오분류) | 엔티티 허브 개념이 시작만 되고 방치됨 |
| `mem.py seek` 검색 로직 | 순수 substring 키워드 스코어링 | 그래프 순회·임베딩 없음, "의미 기반"은 AI가 후속으로 읽고 종합하는 부분뿐 |
| 볼트 전체 태그 커버리지 (오늘 조치 전) | 172개 중 47개(27%) 태그 0개 | → 오늘 소급 태깅으로 해결됨(선행 작업 완료) |

**결론**: 분류(lint)·요약(enrich)·태깅까지는 파이프라인이 실제로 잘 도는데, "노트 간 관계"를 만드는 단계만 설계돼 있고 실행이 거의 안 됐다. 오늘 태깅을 고쳐서 재료(태그)는 갖췄으니, 이제 그 재료로 실제 연결을 만드는 단계가 이 기획의 핵심이다.

## 2. 비전 — 이 프로젝트만의 "킥"

일반 메모앱과 다른 점을 한 줄로: **"저장한 걸 찾아준다"가 아니라 "저장한 것들이 서로 어떻게 연결되는지, 사용자도 몰랐던 걸 보여준다."**

구체적으로 목표하는 사용자 경험:
- "OpenMontage 뭐였지?" → 그 노트 하나만 보여주는 게 아니라 "이것도 관련 있어요: video-use, claude-real-video, DataMagic, PerfectPixel Studio (전부 AI 영상 자동화 도구)"까지 자동으로 붙어 나온다.
- 매주/매달 "이번에 저장한 것 중에 서로 연결되는 클러스터가 생겼어요" 같은, 사용자가 요청하지 않아도 먼저 알려주는 발견(discovery)이 있다.
- 어떤 도구/사람/장소를 두 번 이상 언급하면, 그게 자동으로 "허브 페이지"가 되어 언급될 때마다 쌓인다 (지금은 매번 새 고립 노트만 생김).

## 3. 설계 원칙

1. **점진적, 되돌릴 수 있는 단위로.** improvement-plan.md와 같은 방식 — 작고 명확한 것부터, 서로 다른 파일/함수를 건드려 충돌 없이.
2. **있는 데이터부터 활용.** 임베딩·벡터DB 같은 새 인프라 도입 전에, 이미 있는 태그·frontmatter·MOC로 얼마나 갈 수 있는지 먼저 본다 (Karpathy 원칙 — 볼트가 작으면 단순한 방법이 낫다).
3. **AI가 지어내지 않는다.** 관계 연결도 lint/enrich와 동일 원칙 — 확신 없으면 안 엮는다. 억지 연결은 노이즈만 늘린다.
4. **사용자 확인 없이 대규모로 되돌리기 어려운 변경(대량 이동/병합) 안 함.** 태그 추가·링크 추가처럼 안전한 것부터.

## 4. 컴포넌트 설계

### A. 태그 기반 소급 클러스터링 (Phase 1 — 가장 먼저)
오늘 태깅한 결과를 바로 활용. 같은 태그(예: `agent`, `github`+영상 관련 키워드)를 공유하는 노트들을 찾아 서로 `related`/`[[wikilink]]`로 묶는다.
- 예시(오늘 발견): OpenMontage, video-use, claude-real-video, DataMagic, PerfectPixel Studio — 전부 "AI 영상/이미지 자동화" 계열인데 하나도 안 엮여 있음.
- 방법: 노트 내용을 읽고 실제로 같은 주제인지 판단 후(태그 일치는 후보 필터일 뿐, 최종 판단은 AI가) `## 관련` 섹션에 상호링크 추가.
- 리스크 낮음: 파일 이동·재분류 없음, 링크 추가만.

### B. 엔티티 승격 규칙 명확화 (Phase 2)
지금 ai-lint.md 2단계("반복 등장할 가능성이 있으면 엔티티 생성")가 기준이 애매해서 사실상 안 지켜짐. 명확한 규칙으로 교체:
- **승격 조건**: 같은 도구/사람/프로젝트/장소가 **2회 이상** 언급되면 자동으로 엔티티 허브 페이지 승격 (학습 규칙 ③d의 "2회 확정 시 자동 승격" 패턴과 동일한 사고방식 재사용).
- 승격 시: `40_Entities/{Type}/{이름}.md` 생성, 기존 언급 노트들을 전부 역링크로 모음, 이후 같은 대상이 또 언급되면 새 노트가 아니라 이 허브에 이벤트를 추가.
- 1회성 언급(오늘 만든 OpenMontage 같은 것)은 지금처럼 개별 노트로 충분 — 모든 걸 엔티티로 만들 필요 없음(노이즈 방지).

### C. Lint 시점 관계 탐지 강화 (Phase 2, ai-lint.md 2단계 보강)
새 raw note 처리 시 지금은 "엔티티 페이지 있는지"만 본다. 추가로:
- `mem.py seek`로 같은 태그/유사 키워드를 가진 기존 노트를 후보로 조회 (이미 있는 함수 재사용, 새 인프라 불필요)
- 후보 중 실제로 관련 있다고 판단되면 신규 노트 ↔ 기존 노트 상호링크
- 이건 매 lint마다 자동으로 도는 절차라, 시간이 지날수록 그래프가 스스로 촘촘해짐 (지금처럼 매번 고립 노트만 쌓이는 것과 대조)

### D. 백링크 인식 (Phase 3)
지금 파이프라인은 "이 노트가 무엇을 링크하는지"만 알고 "무엇이 이 노트를 링크하는지"(backlink)는 전혀 추적 안 함 (옵시디언 자체 그래프 뷰엔 있지만 AI 파이프라인엔 없음).
- `mem.py`에 경량 백링크 인덱스 함수 추가 (전체 볼트 스캔해서 `path → [이 파일을 링크하는 파일들]` 맵 생성, seek/entity 페이지에서 재사용)
- 엔티티 허브 페이지가 "이 항목을 언급한 노트 전체"를 항상 최신으로 보여줄 수 있게 됨

### E. Seek 검색 그래프 확장 (Phase 3~4, 코드 변경)
`scripts/mem.py::seek`에 1-hop 확장 추가:
- 키워드 스코어링으로 상위 후보를 찾은 뒤, 그 노트가 링크하는 파일 + 그 노트를 링크하는 파일(백링크, D 완료 후)까지 후보 풀에 추가
- AI가 답변 종합할 때 이 확장된 후보까지 같이 읽음 → "관련 기억" 섹션이 지금처럼 AI가 감으로 추측하는 게 아니라 실제 그래프 구조에서 나옴

### F. 큐레이터형 "발견한 연결" 다이제스트 (2026-07-15 상세 설계 추가)

**문제의식**: 그냥 "이번 주 저장한 것 묶어서 보여주기"는 기계적이다. "GitHub 저장소 5개 저장하셨네요" 같은 건 파일명만 봐도 이미 아는 사실이라 발견이 아니다. 진짜 큐레이터/비서라면 사용자 본인도 명확히 의식 못 했던 패턴 — 반복되는 관심사, 우연이 아닌 것 같은 취향, 시간이 지나야 보이는 흐름 — 을 짚어준다. 이 절이 이번 기획의 핵심 차별화 지점이라 가장 상세히 설계한다.

#### F-1. "기계적 나열"과 "진짜 발견"을 가르는 기준

| 기계적 (하지 않을 것) | 진짜 발견 (목표) |
|---|---|
| 이번 주 저장한 것만 봄 | **여러 주/달에 걸쳐 반복되는 패턴**을 봄 (한 주만 보면 우연과 진짜 관심사를 구분 못 함) |
| 태그 하나로 묶어서 그대로 나열 | 왜 흥미로운지 **이유**를 붙임 (반복성, 문제의식의 공통점, 시점 등) |
| 매주 반드시 뭔가 보고 | **기준 미달이면 이번 주는 조용히 넘어감** ("매주 뭔가 있다"고 우기는 게 오히려 신뢰를 깎는다) |
| 단정적으로 말함 ("~하시는군요") | **관찰+질문 형태**로 제안 ("~인 것 같은데, 맞나요?") — 틀려도 무례하지 않게 |
| 카테고리 안에서만 묶음 (save끼리만) | **타입을 넘나드는 연결**을 우선한다 (예: 저장한 도구 + 관련 task가 이어지는 경우가 훨씬 흥미로움) |

#### F-2. 신호(무엇을 흥미롭다고 판단할 것인가)

아래 신호를 조합해 "이번 주 보여줄 가치가 있는가"를 스코어링한다. 전부 이미 있는 데이터(태그·날짜·raw 원문)로 계산 가능, 새 인프라 불필요.

1. **시간에 걸친 반복** — 같은 태그/주제가 서로 다른 주(week)에 걸쳐 등장 (이번 주만 3개면 우연일 수 있지만, 지난달과 이번 주에 나눠 등장하면 진짜 지속적 관심사)
2. **타입 교차** — 서로 다른 memory_type(예: save + task, song + journal)이 같은 주제로 이어짐 — 카테고리 안에서의 반복보다 훨씬 신호가 강함
3. **원문의 감정 신호** — raw 텍스트에 "반드시 써야함", "완전 좋다", "대박" 같은 강한 어조가 있으면 단순 북마크보다 우선순위를 높임 (사실 기반 — 실제 텍스트에 있는 표현만 근거로 삼는다, 지어내지 않음)
4. **누적량** — 클러스터 크기가 일정 수(예: 3개) 이상일 때만 후보로 삼음 (2개는 우연일 확률이 높음)

스코어 = (반복 주 수 × 3) + (타입 교차 시 +5) + (감정 신호 매치 시 +2) + (클러스터 크기, 최대 +3). 상위 1~3개만 채택 — **많이 보여주는 게 아니라 적게, 확실한 것만 보여주는 게 큐레이터의 핵심**.

#### F-3. 실제 발견 예시 (오늘 데이터로 시뮬레이션)

```
💡 이번 주 발견한 것

최근 두 달 사이 "AI가 대화 사이 기억을 어떻게 유지할까"라는 주제의 프로젝트를
4번이나 저장하셨어요 — pmb, engram, Hindsight, ai-memory-vault. 저장 시점이
6월 말부터 꾸준히 이어지는 걸 보면 우연이 아니라 계속 눈여겨보고 계신 주제
같아요. 혹시 직접 관련 프로젝트를 만들어보실 계획이 있으신가요? 원하시면
4개의 접근 방식 차이를 비교한 노트를 만들어드릴게요.

📄 근거: [[pmb]] (06-14) · [[engram]] (07-12) · [[Hindsight]] (06-?) · [[ai-memory-vault]] (07-09)
```

기계적 버전과 비교: "이번 주 AI 메모리 관련 노트 4개 저장했습니다" — 같은 데이터인데 "왜 흥미로운지"(지속성), "무엇을 의미할 수 있는지"(직접 만들 계획?), "다음 행동 제안"(비교 노트 제안)이 빠지면 그냥 알림이지 발견이 아니다.

#### F-4. 톤 원칙

- 확신 있게 단정하지 않는다 — "~인 것 같아요", "혹시 ~신가요?" 톤 유지 (이 프로젝트 전체의 "지어내지 않는다" 원칙과 동일선상)
- 매주 반드시 뭔가 보고할 필요 없음 — 기준 미달이면 "이번 주는 발견한 거 없어요"도 정답
- private/sensitivity 표시된 노트가 섞인 클러스터는 다이제스트에서 제외 (민감정보 노출 방지)
- 발견 뒤에는 항상 구체적 노트 경로를 근거로 남긴다 (막연한 인상이 아니라 출처 있는 관찰)

#### F-5. 아키텍처 배치

- 기존 `weeklyDigest`(interval 7일, hour 9)와 같은 트리거를 공유하되, 실제 클러스터 탐지+문구 작성은 결정적 스크립트가 아니라 **AI 판단이 필요한 신규 job 타입**(`discover`)으로 큐에 등록 → 23:00 AI 배치에서 lint/enrich와 같은 방식으로 처리
- 탐지 스크립트(태그·날짜 스캔, 스코어링)는 `mem.py`에 결정적 함수로 추가 가능 (새 인프라 없음) — 결과 후보 리스트만 스코어와 함께 job payload에 담고, "왜 흥미로운지 문구화"는 AI가 담당 (lint/enrich와 동일한 역할 분담: 결정적 계산은 스크립트, 판단·문장은 AI)

### G. 중복/오분류 자동 탐지
오늘 우연히 발견한 "코깍 사용설명서" 중복(20_Records/Maintenance ↔ 40_Notes/Things) 같은 사례를 `ai-doctor.md`/`ai-repair.md` 루틴에 정기 점검 항목으로 추가

## 6. 트레이드오프 정리

- **자동 연결 vs 노이즈**: 너무 적극적으로 엮으면 "관련 기억"이 소음이 되고, 너무 보수적이면 지금처럼 그래프가 텅 빈다. → 승격 조건(2회 이상)과 AI의 "확신 없으면 안 엮는다" 원칙으로 균형.
- **임베딩 검색 도입 여부**: 이번 기획엔 포함 안 함. 태그+그래프 확장으로 충분한지 먼저 보고, 부족하면 그 다음 단계로 고려 (`docs/improvement-plan.md`의 P5-3단계, "노트 수천 개 될 때 착수"와 같은 결의 판단).
- **소급 작업 범위**: 139개 Saves 전체를 한 번에 엔티티화하면 리스크 크다 → Phase 3에서 소규모 샘플부터 시작해 확대.
