# AI Discover Prompt — 큐레이터형 "발견한 연결" 다이제스트

Goal: `scripts/discover.py`(결정적, 태그 클러스터 스코어링)가 뽑아준 후보 중에서, **진짜로
흥미로운 발견**만 골라 한국어로 짧게 소개한다. 목표는 "저장한 걸 찾아준다"가 아니라
"저장한 것들이 서로 어떻게 연결되는지, 사용자도 몰랐던 걸 먼저 보여주는" 것.
설계 배경·톤 원칙 전체는 `docs/graph-layer-plan.md` § F 참조.

## 절대 원칙

- **지어내지 않는다.** 모든 관찰은 실제 노트 내용·날짜·태그에서만 근거를 가져온다.
- **매주 반드시 뭔가 보고할 필요 없다.** 아래 기준을 통과하는 후보가 없으면 "이번 주는 발견한
  거 없음"으로 job을 종료한다 (Telegram 발송 안 함). 매주 억지로 뭔가 찾아내면 오히려 신뢰가
  깎인다 — 침묵도 정답이다.
- **적게, 확실한 것만.** 후보가 여러 개 통과해도 실제로는 1개(최대 2개)만 골라 보낸다.
- 노트를 이동·재분류하거나 frontmatter를 바꾸지 않는다. 이 job이 하는 일은 **wikilink 추가(선택)와
  Telegram 발송뿐**이다.

## 처리 단계

### 1단계: 후보 다시 조회 (job 생성 시점 스냅샷 신뢰하지 않음)

```bash
python3 scripts/mem.py discover --top-n 8
```

job payload의 `raw_text`(예: "discover: 후보 3건, 최고점 23 (태그: 문서)")는 트리거 사실만 알려줄
뿐이니, 실제 판단은 이 명령을 다시 실행해서 **지금 시점 데이터**로 한다.

### 2단계: 진짜 흥미로운 후보 걸러내기 (여기가 핵심 — 스크립트가 못 하는 부분)

`discover.py`는 태그로만 묶어서 점수를 매긴 것이지, 그 태그가 **진짜 관심사/주제**인지
**그냥 형식·속성 태그**(예: `문서`, `마크다운`, `macOS`, `크롬확장`, `한국어`, `무료`, `로컬`,
`개발자도구`, `UI/UX` 같은 — 내용이 아니라 형태·플랫폼을 나타내는 태그)인지는 구분하지 못한다.

각 후보에 대해:
1. 묶인 노트들을 실제로 Read로 열어 내용을 확인한다
2. 정말 같은 **주제/문제의식**을 공유하는지 판단한다 (형식만 같고 내용은 무관하면 탈락)
3. 판단 기준(우선순위):
   - 여러 주/달에 걸쳐 반복되는가 (한 주에 몰려있으면 우연일 수 있음 — `weeks_spanned` 참고하되 직접 날짜도 확인)
   - 서로 다른 memory_type을 넘나드는가 (`type_diversity` > 1이면 특히 흥미로움 — 저장한 도구와 할일이 이어지는 식)
   - 원문에 강한 어조(`emotion_hits` > 0)가 있는가 — 실제 텍스트를 확인해 진짜 반응인지 확인
4. 통과하는 후보가 없으면 → 4단계로 건너뛰어 "발견 없음"으로 종료
5. 통과하는 후보가 여러 개면 점수 높은 순으로 최대 2개만 선택

### 3단계: 관찰 + 질문 형태로 문구 작성 (한국어, 톤 원칙)

- **단정하지 않는다**: "~하시는군요" 대신 "~인 것 같아요", "~신가요?" 톤
- 왜 흥미로운지 이유를 반드시 포함 (지속성/타입 교차/감정 신호 중 실제로 근거가 된 것)
- 가능하면 다음 행동을 가볍게 제안 (예: "비교 노트 만들어드릴까요?") — 강요 톤 아님
- 근거 노트 경로를 항상 함께 남긴다 (막연한 인상이 아니라 출처 있는 관찰)

형식 (`docs/graph-layer-plan.md` F-3 예시 참고):

```
💡 이번 주 발견한 것

{2~4문장: 무엇을 발견했는지, 왜 흥미로운지(지속성/타입교차/감정신호), 가벼운 제안}

📄 근거: [[노트1]] ({날짜}) · [[노트2]] ({날짜}) · ...
```

후보가 2개 선택됐으면 위 블록을 두 번 반복(사이에 구분선 `━━━━━━━━━━`).

### 4단계: 발견 없음일 때

Telegram 발송 없이 job만 완료 처리한다. job result에 `"discovered": false, "reason": "..."` 를
남겨 다음에 왜 스킵됐는지 추적 가능하게 한다.

### 5단계: 선택적 wikilink 추가

발견을 실제로 발송한 클러스터는, 근거로 쓴 노트들 사이에 아직 상호링크가 없으면
`## 관련` 섹션에 추가해도 된다 (Phase 1 클러스터링과 동일한 방식 — `ai-lint.md` 참고).
이동·재분류·frontmatter 변경은 하지 않는다.

### 6단계: Telegram 발송 + job 완료

job의 `chat_id`가 있으면(없으면 `memory-config.json`의 `telegram.allowedUserIds[0]`) 3단계 문구를
발송한다. 발송 후:

```bash
python3 scripts/jobs.py set-status <job_id> done --note "discover: N건 발송" \
  --result-json '{"discovered": true, "clusters": ["<태그1>", "<태그2>"]}'
```

발견 없음이면:

```bash
python3 scripts/jobs.py set-status <job_id> done --note "discover: 이번 주 발견 없음" \
  --result-json '{"discovered": false, "reason": "후보 없음 또는 전부 형식성 태그로 판단"}'
```

## 주의사항

- `sensitivity: private` 노트가 섞인 클러스터는 `discover.py`가 이미 제외하지만, 개별 노트 내용
  중 민감해 보이는 부분이 있으면 문구에서 언급하지 않는다
- 같은 클러스터를 반복해서 매주 보내지 않는다 — 지난 발견과 겹치면 스킵하거나 "지난번과 이어지는
  발전"으로만 짧게 언급 (예: "지난주 말씀드린 ~ 관심사, 이번 주에도 또 나왔어요")
